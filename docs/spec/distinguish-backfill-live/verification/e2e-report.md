## Findings

### 1. Feature works exactly as specified

Every observable behavior matched the spec. The headline win: starting `watch`
on a folder of pre-existing sessions now produces zero LLM cost. Sessions are
still ingested for searchability; summarization is correctly deferred to
either (a) the user resuming the session, which promotes it to live and the
watermark gate then evaluates, or (b) explicit `summarize <id>` / `--all`.

### 2. Watermark gate works on live re-summarization too

Scenario 3 confirmed the gate fires not only on backfill but on continued live
sessions: small appends (delta < 5) are skipped, larger appends (delta â‰Ą 5)
trigger re-summarization. This addresses the staleness concern that prompted
adding the watermark in the first place.

### 3. `summarize` command UX is correct

Confirmation text matches the spec character-for-character. Exit codes are
unambiguous (0 success / 1 partial failure / 2 usage). `--yes`, `--force`,
`--since` (not exercised here, covered by unit tests), `--all`, and the bare
positional id all work.

### 4. Pre-existing condition: `summarization_runs` table stayed empty

The `summarization_runs` table received zero rows during this run. Investigation
showed `recordSummarizationRun` is referenced nowhere in the CLI on this
branch â€” the feature that populates it lives in untracked files in `main`'s
working tree (`packages/server/src/routes/summarization-runs.ts`,
`packages/cli/src/commands/runs.ts`). **Not a regression from this branch.**
Worth noting because the spec's verification matrix mentioned this table, and
a reader might otherwise be confused.
