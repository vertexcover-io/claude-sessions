# PROMPT log — claude-sessions

This subdirectory was generated by Claude Code. The content here captures the conversation arc, scope evolution, key architectural decisions, and the final shape of the project.

## Model & agent

- Model: Claude Opus 4.7 (1M context)
- Agent: Claude Code orchestrator skill running brainstorm → planning → coder pipeline (8 phases, TDD with vitest)
- Spec/plan/phase artifacts: [`../docs/spec/claude-session-finder/`](../docs/spec/claude-session-finder/)

## Original ask

> "Build something that lets me find old Claude Code sessions by what they
> were about — not just by date or filename. I want to ask 'where was
> that session where I debugged the auth bug last month?' and get an
> answer."

The user also referenced a Claude.ai-style screenshot as the visual target for the dashboard — a list of sessions, each with a generated title and a one-line summary, searchable by content.

## Scope evolution

### v1: personal local-only

- CLI that indexes `~/.claude/projects/**/*.jsonl`
- SQLite for storage
- Local web app (Tauri considered) for browsing
- `claude -p` for LLM-generated summaries
- No server, no auth, no upload

### v2: multi-machine cloud sync (scope expansion)

User pivoted: "I work across 3 machines. I want my sessions queryable from all of them, including from inside Claude Code via MCP. And I want my teammates to see my sessions too."

- Server with email/password auth, JWT tokens (CLI/web/MCP audiences)
- Per-repo opt-in privacy: `claude-sessions enable <path>` only syncs that repo's sessions (REQ-013)
- Per-session privacy via `~/.claude-sessions/sessions/<sessionId>.private` sidecar (REQ-040)
- Redaction at rest: AWS keys, GitHub tokens, OpenAI keys, JWTs, env-vars, high-entropy strings — applied on the CLI AND again on the server (defense-in-depth, REQ-034)
- Postgres + pgvector for hybrid (FTS + cosine) search
- MCP server so Claude Code itself can query the index (six tools, REQ-047)

### v3: simplification (the cuts)

User pushed back on architectural complexity:

- **Drop Tauri/desktop shell.** "Just a web SPA served by the API server is fine."
- **Drop SQLite.** "Postgres on the server is the only persistence." Local CLI keeps only `~/.claude-sessions/state.json` (per-file byte offsets) — not a cache.
- **No worker, no Redis, no outbox table, no background job runner.** All write-side work (redact, persist, embed) happens inline in the request handler that triggered it (REQ-038, REQ-063). Summary upload returns only after both the summary row and embedding row are persisted in one transaction.
- **Store the raw JSONL blob as `bytea` in Postgres**, not S3. Keeps deploy to a single Postgres + Node container (REQ-061, REQ-062). 100MB cap.
- **Inline embedding in the summary handler** rather than a separate `/embed` route or background backfill.

### v4: feature additions

- **Fork from checkpoint** (REQ-050–REQ-056): UI button on any event copies a `claude-sessions fork <id> --until <uuid> --cwd <path>` command. CLI fetches the blob, truncates to that event, rewrites every line's `cwd`, assigns a new `sessionId`, sets first line's `parentUuid: null`, writes `~/.claude/projects/<encoded-cwd>/<new-sessionId>.jsonl`, and prints the resume command. Refuses to overwrite (EDGE-022).
- **User-set session names** (REQ-059, REQ-060): `claude-sessions name <id> "<name>"` overrides the LLM-generated title. Display order: user `name` → LLM `title` → `Session <prefix>`.

## Architecture decisions (one-line each)

| Decision | Why |
|---|---|
| Single Postgres, no SQLite | One backup target; pgvector only lives server-side anyway |
| pgvector inline embedding (not async) | Removes outbox/worker/Redis; summary handler returns when both rows are committed (REQ-038) |
| `bytea` blob in Postgres (not S3) | Single-container deploy; 100MB cap is plenty for transcripts (REQ-061) |
| Single-process Hono server | No worker tier; static SPA served from same process (REQ-063) |
| Per-repo opt-in | Zero-cost privacy default; CLI never auto-uploads anything until `enable` runs |
| Per-session sidecar privacy flag | User can withdraw without running a CLI command — drop a file and the watcher PATCHes the server next tick |
| MCP on same Hono process | One token-issuance flow; one transport; tools share `searchInternal` and `sessions-internal` helpers with the REST routes |
| Argon2id + JWT (`jose`) with audiences | Audiences (`cli`, `web`, `mcp`) prevent cross-context token reuse |
| Hybrid FTS + vector with RRF | FTS for keywords, vectors for semantics; RRF k=60 fuses the two ranked lists deterministically |
| Defense-in-depth redaction | CLI redacts before upload, server redacts again at write time so a misconfigured client cannot poison storage |

## What the user explicitly asked vs decided

| Topic | User asked | Decided |
|---|---|---|
| Storage backend | "I don't want sqlite + postgres" | Postgres only; CLI `state.json` is byte offsets, not a cache |
| Job runner | "no worker, no redis" | Inline request handlers (REQ-063) |
| Blob storage | "S3 feels heavy" | `bytea` in Postgres (100MB cap) |
| Embedding model | "OpenAI is fine for v0" | `text-embedding-3-small` default (1536-dim); `bge` ONNX path stubbed |
| Auth | "skip GitHub OAuth for now" | email + argon2id password only in v0 |
| Adapter scope | "Claude only for v0" | `adapter-claude` only; canonical schema in `@claude-sessions/core` is forward-compatible to other agents |
| Web stack | (no preference) | Vite + React 18 + Tailwind v4 + TanStack Query/Virtual + Shiki |
| Server framework | (no preference) | Hono on Node 22 |
| Desktop shell | "do I really need Tauri?" | No — web SPA only |
| MCP | "must work from inside Claude Code" | 6 MCP tools mounted on the same Hono process at `/mcp/:token` |

## Final shape

- **6 packages** in a Turborepo monorepo:
  - `@claude-sessions/core` — canonical types, pricing, redaction, git repo detection
  - `@claude-sessions/adapter-claude` — JSONL streaming → canonical events
  - `@claude-sessions/cli` — `claude-sessions` binary (12 commands)
  - `@claude-sessions/server` — Hono + Postgres + pgvector + MCP transport
  - `@claude-sessions/web` — React SPA, served from the API server's static handler
  - `@claude-sessions/test-config` — shared vitest config
- **194 tests** across 8 phases (phase-0 skeleton → phase-7 polish)
- **One Postgres migration** (`packages/server/src/db/migrations/0001_init.sql`) creating users, repos, sessions, events, summaries, embeddings, session_blobs, session_pr_links, audit_log, plus the `summaries_fts_text` immutable function and the HNSW cosine index
- **Single docker-compose service** (`pgvector/pgvector:pg16` on host port 5433) — no Redis, no S3, no worker
